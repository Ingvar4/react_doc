`useInsertionEffect` позволяет вставлять элементы в DOM до срабатывания любых layout Effects.

```js
useInsertionEffect(setup, dependencies?)
```

---

## Reference {/*reference*/}

### `useInsertionEffect(setup, dependencies?)` {/*useinsertioneffect*/}

Вызывайте `useInsertionEffect` на верхнем уровне компонента, чтобы вставить стили до срабатывания любых Effects, которым может понадобиться прочитать layout:

```js
import { useInsertionEffect } from 'react';

// Inside your CSS-in-JS library
function useCSS(rule) {
  useInsertionEffect(() => {
    // ... inject <style> tags here ...
  });
  return rule;
}
```

#### Параметры {/*parameters*/}

* `setup`: Функция с логикой вашего Effect. Функция setup также может при необходимости вернуть функцию *cleanup*. Когда ваш компонент добавляется в DOM, но до срабатывания любых layout Effects, React выполнит вашу функцию setup. После каждого повторного рендера с изменившимися зависимостями React сначала выполнит функцию cleanup (если вы её предоставили) со старыми значениями, а затем выполнит вашу функцию setup с новыми значениями. Когда ваш компонент удаляется из DOM, React выполнит вашу функцию cleanup.

* **необязательный** `dependencies`: Список всех реактивных значений, на которые есть ссылки внутри кода `setup`. Реактивные значения включают props, state, а также все переменные и функции, объявленные прямо внутри тела компонента. Если ваш линтер [настроен для React](/learn/editor-setup#linting), он проверит, что каждое реактивное значение корректно указано в качестве зависимости. Список зависимостей должен иметь постоянное число элементов и быть записан inline, например `[dep1, dep2, dep3]`. React будет сравнивать каждую зависимость с её предыдущим значением, используя алгоритм сравнения [`Object.is`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Object/is). Если вы вообще не укажете зависимости, ваш Effect будет запускаться повторно после каждого повторного рендера компонента.

#### Возвращаемое значение {/*returns*/}

`useInsertionEffect` возвращает `undefined`.

#### Предостережения {/*caveats*/}

* Effects выполняются только на клиенте. Они не выполняются во время серверного рендеринга.
* Вы не можете обновлять state изнутри `useInsertionEffect`.
* К моменту вызова `useInsertionEffect` refs ещё не присоединены.
* `useInsertionEffect` может выполниться либо до, либо после обновления DOM. Не стоит полагаться на то, что DOM обновлён в какой-то конкретный момент.
* В отличие от других типов Effects, которые сначала вызывают cleanup для каждого Effect, а затем setup для каждого Effect, `useInsertionEffect` вызывает cleanup и setup по одному компоненту за раз. В результате получается "перемежение" функций cleanup и setup.

## Usage {/*usage*/}

### Внедрение динамических стилей из CSS-in-JS библиотек {/*injecting-dynamic-styles-from-css-in-js-libraries*/}

Традиционно React-компоненты стилизуют обычным CSS.

```js
// In your JS file:
<button className="success" />

// In your CSS file:
.success { color: green; }
```

Некоторые команды предпочитают писать стили напрямую в JavaScript-коде вместо отдельных CSS-файлов. Обычно для этого нужен CSS-in-JS library или инструмент. Есть три распространённых подхода к CSS-in-JS:

1. Статическое извлечение в CSS-файлы с помощью компилятора
2. Inline styles, например `<div style={{ opacity: 1 }}>`
3. Внедрение тегов `<style>` во время выполнения

Если вы используете CSS-in-JS, мы рекомендуем сочетание первых двух подходов (CSS-файлы для статических стилей, inline styles для динамических стилей). **Мы не рекомендуем внедрение тегов `<style>` во время выполнения по двум причинам:**

1. Внедрение во время выполнения заставляет браузер гораздо чаще пересчитывать стили.
2. Внедрение во время выполнения может быть очень медленным, если оно происходит в неподходящий момент в жизненном цикле React.

Первую проблему решить нельзя, но `useInsertionEffect` помогает решить вторую.

Вызывайте `useInsertionEffect`, чтобы вставить стили до срабатывания любых layout Effects:

```js 
// Inside your CSS-in-JS library
let isInserted = new Set();
function useCSS(rule) {
  useInsertionEffect(() => {
    // As explained earlier, we don't recommend runtime injection of <style> tags.
    // But if you have to do it, then it's important to do in useInsertionEffect.
    if (!isInserted.has(rule)) {
      isInserted.add(rule);
      document.head.appendChild(getStyleForRule(rule));
    }
  });
  return rule;
}

function Button() {
  const className = useCSS('...');
  return <div className={className} />;
}
```

Подобно `useEffect`, `useInsertionEffect` не выполняется на сервере. Если вам нужно собирать, какие CSS-правила были использованы на сервере, вы можете делать это во время рендеринга:

```js 
let collectedRulesSet = new Set();

function useCSS(rule) {
  if (typeof window === 'undefined') {
    collectedRulesSet.add(rule);
  }
  useInsertionEffect(() => {
    // ...
  });
  return rule;
}
```

#### Чем это лучше, чем внедрение стилей во время рендеринга или useLayoutEffect? {/*how-is-this-better-than-injecting-styles-during-rendering-or-uselayouteffect*/}

Если вы вставляете стили во время рендеринга и React обрабатывает [неблокирующее обновление,](/reference/react/useTransition#perform-non-blocking-updates-with-actions) браузер будет пересчитывать стили на каждом кадре, пока рендерится дерево компонентов, и это может быть **крайне медленно.**

`useInsertionEffect` лучше, чем внедрение стилей через [`useLayoutEffect`](/reference/react/useLayoutEffect) или [`useEffect`](/reference/react/useEffect), потому что он гарантирует, что к моменту запуска других Effects в ваших компонентах теги `<style>` уже будут вставлены. Иначе расчёты layout в обычных Effects будут неверными из-за устаревших стилей.

</Callout>
