
## Справка {/*reference*/}

### `useMemo(calculateValue, dependencies)` {/*usememo*/}

Вызовите `useMemo` на верхнем уровне компонента, чтобы кэшировать вычисление между повторными рендерами:
```js
import { useMemo } from 'react';

function TodoList({ todos, tab }) {
  const visibleTodos = useMemo(
    () => filterTodos(todos, tab),
    [todos, tab]
  );
  // ...
}
```

#### Параметры {/*parameters*/}

* `calculateValue`: Функция, которая вычисляет значение, которое вы хотите кэшировать. Она должна быть pure, не должна принимать аргументов и должна возвращать значение любого типа. React вызовет эту функцию при первом рендере. При следующих рендерах React снова вернёт то же значение, если `dependencies` не изменились с прошлого рендера. В противном случае React вызовет `calculateValue`, вернёт её результат и сохранит его, чтобы использовать позже.

* `dependencies`: Список всех reactive values, на которые ссылается код внутри `calculateValue`. Reactive values включают props, state и все переменные и функции, объявленные прямо внутри тела компонента. Если ваш linter [настроен для React](/learn/editor-setup#linting), он проверит, что каждое reactive value корректно указано как dependency. Список dependencies должен содержать постоянное число элементов и быть записан inline, например `[dep1, dep2, dep3]`. React будет сравнивать каждую dependency с её предыдущим значением с помощью сравнения [`Object.is`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Object/is).

#### Возвращаемое значение {/*returns*/}

При первом рендере `useMemo` возвращает результат вызова `calculateValue` без аргументов.

При последующих рендерах он либо вернёт уже сохранённое значение с прошлого рендера (если dependencies не изменились), либо снова вызовет `calculateValue` и вернёт результат, который она вернёт.

#### Предостережения {/*caveats*/}

* `useMemo` — это Hook, поэтому вы можете вызывать его только **на верхнем уровне вашего компонента** или ваших собственных Hooks. Вы не можете вызывать его внутри циклов или условий. Если вам это нужно, вынесите новый компонент и перенесите state в него.
* В Strict Mode React **вызовет вашу функцию вычисления дважды**, чтобы [помочь вам найти случайные impurities.](#my-calculation-runs-twice-on-every-re-render) Это поведение только для разработки и не влияет на production. Если ваша функция вычисления pure (как и должна быть), это не повлияет на логику. Результат одного из вызовов будет проигнорирован.
* React **не будет выбрасывать кэшированное значение без конкретной причины.** Например, в development React сбрасывает кэш, когда вы редактируете файл компонента. И в development, и в production React сбросит кэш, если компонент suspend'ится во время начального mount. В будущем React может добавить новые возможности, которые будут использовать такой сброс кэша — например, если появится встроенная поддержка virtualized lists, будет логично сбрасывать кэш для элементов, которые выходят за пределы viewport виртуализированной таблицы. Это нормально, если вы используете `useMemo` только как performance optimization. В противном случае более уместны [state variable](/reference/react/useState#avoiding-recreating-the-initial-state) или [ref](/reference/react/useRef#avoiding-recreating-the-ref-contents).

<Callout>

Кэширование возвращаемых значений таким способом также называется [*memoization*,](https://en.wikipedia.org/wiki/Memoization) поэтому этот Hook и называется `useMemo`.

</Callout>

## Использование {/*usage*/}

### Пропуск дорогостоящих повторных вычислений {/*skipping-expensive-recalculations*/}

Чтобы кэшировать вычисление между повторными рендерами, заверните его в вызов `useMemo` на верхнем уровне компонента:
```js 
import { useMemo } from 'react';

function TodoList({ todos, tab, theme }) {
  const visibleTodos = useMemo(() => filterTodos(todos, tab), [todos, tab]);
  // ...
}
```
В `useMemo` нужно передать две вещи:

1. функцию вычисления, которая не принимает аргументов, как `() =>`, и возвращает то, что нужно было вычислить.
2. список dependencies, включающий каждое значение из вашего компонента, которое используется внутри вычисления.

При первом рендере значение, которое вы получите из `useMemo`, будет результатом вызова вашей функции вычисления.

При каждом следующем рендере React будет сравнивать dependencies с dependencies из прошлого рендера. Если ни одна из них не изменилась (по сравнению с [`Object.is`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Object/is)), `useMemo` вернёт то значение, которое было уже вычислено ранее. Иначе React заново выполнит вычисление и вернёт новое значение.

Иными словами, `useMemo` кэширует результат вычисления между рендерами, пока dependencies не изменятся.

**Давайте разберём пример, чтобы увидеть, когда это полезно.**

По умолчанию React будет заново выполнять всё тело компонента при каждом повторном рендере. Например, если `TodoList` обновит state или получит новые props от родителя, функция `filterTodos` выполнится снова:
```js {2}
function TodoList({ todos, tab, theme }) {
  const visibleTodos = filterTodos(todos, tab);
  // ...
}
```
Обычно это не проблема, потому что большинство вычислений очень быстрые. Но если вы фильтруете или преобразуете большой массив, либо выполняете дорогое вычисление, вам может понадобиться не делать это снова, если данные не изменились. Если и `todos`, и `tab` такие же, как и при прошлом рендере, обёртка вычисления в `useMemo`, как выше, позволит вам переиспользовать уже рассчитанный `visibleTodos`.

Такой тип кэширования называется *[memoization.](https://en.wikipedia.org/wiki/Memoization)*

<Callout>

**Использовать `useMemo` стоит только как performance optimization.** Если код без него не работает, сначала найдите и исправьте основную проблему. После этого можно добавить `useMemo`, чтобы улучшить производительность.

</Callout>

<Callout type='idea'>

#### Как понять, дорогое ли вычисление? {/*how-to-tell-if-a-calculation-is-expensive*/}

В общем случае, если вы не создаёте и не проходите по тысячам объектов, вычисление, скорее всего, не является дорогим. Если хотите убедиться, можно добавить `console.log` или замер времени в кусок кода:
```js {1,3}
console.time('filter array');
const visibleTodos = filterTodos(todos, tab);
console.timeEnd('filter array');
```
Выполните измеряемое действие (например, ввод текста в input). Затем вы увидите в консоли что-то вроде `filter array: 0.15ms`. Если суммарное время оказывается заметным (скажем, `1ms` или больше), возможно, есть смысл мемоизировать это вычисление. Для проверки можно обернуть вычисление в `useMemo` и посмотреть, уменьшится ли общее время для этого взаимодействия:

```js
console.time('filter array');
const visibleTodos = useMemo(() => {
  return filterTodos(todos, tab); // Пропускается, если todos и tab не изменились
}, [todos, tab]);
console.timeEnd('filter array');
```

`useMemo` не ускорит *первый* рендер. Он помогает только пропускать лишнюю работу при обновлениях.

Помните, что ваш компьютер, скорее всего, быстрее, чем компьютеры ваших пользователей, поэтому стоит тестировать производительность с искусственным замедлением. Например, в Chrome есть опция [CPU Throttling](https://developer.chrome.com/blog/new-in-devtools-61/#throttling).

Также важно понимать, что замеры в development не дают самых точных результатов. (Например, когда [Strict Mode](/reference/react/StrictMode) включён, вы увидите, что каждый компонент рендерится дважды, а не один раз.) Чтобы получить более точные цифры, собирайте приложение для production и тестируйте его на устройстве, похожем на устройства ваших пользователей.

</Callout>

<Callout type='idea'>

#### Нужно ли добавлять `useMemo` везде? {/*should-you-add-usememo-everywhere*/}

Если ваше приложение похоже на этот сайт, и большинство взаимодействий крупные (например, замена страницы или целого раздела), мемоизация обычно не нужна. С другой стороны, если ваше приложение больше похоже на графический редактор, и большинство взаимодействий мелкие (например, перемещение фигур), мемоизация может быть очень полезна.

Оптимизация с `useMemo` действительно полезна только в нескольких случаях:

- Вычисление, которое вы помещаете в `useMemo`, заметно медленное, и его dependencies редко меняются.
- Вы передаёте результат как prop в компонент, обёрнутый в [`memo`.](/reference/react/memo) Вы хотите пропускать повторный рендер, если значение не изменилось. Мемоизация позволяет компоненту перерендериваться только когда dependencies не совпадают.
- Передаваемое вами значение позже используется как dependency какого-то Hook. Например, от него может зависеть другой результат `useMemo`. Или вы зависите от этого значения в [`useEffect`.](/reference/react/useEffect)

В других случаях пользы от обёртки вычисления в `useMemo` нет. И большого вреда тоже нет, поэтому некоторые команды решают не разбирать каждый случай отдельно и мемоизировать как можно больше. Минус такого подхода в том, что код становится менее читаемым. Кроме того, не всякая мемоизация эффективна: одного значения, которое "всегда новое", достаточно, чтобы сломать мемоизацию целого компонента.

**На практике можно сильно уменьшить необходимость в мемоизации, если следовать нескольким принципам:**

1. Когда компонент визуально оборачивает другие компоненты, пусть он [принимает JSX как children.](/learn/passing-props-to-a-component#passing-jsx-as-children) Тогда, если wrapper-компонент обновляет собственный state, React понимает, что его children не нужно перерендеривать.
1. Предпочитайте локальный state и не [поднимайте state выше](/learn/sharing-state-between-components) без необходимости. Например, не храните временный state, такой как формы или состояние hover, в верхушке дерева или в глобальной state-библиотеке.
1. Держите [логику рендера pure.](/learn/keeping-components-pure) Если повторный рендер компонента вызывает проблему или заметный визуальный артефакт, это баг в компоненте! Исправьте баг, а не добавляйте мемоизацию.
1. Избегайте [ненужных Effects, которые обновляют state.](/learn/you-might-not-need-an-effect) Большинство проблем с производительностью в React-приложениях вызываются цепочками обновлений, которые начинаются в Effects и заставляют компоненты рендериться снова и снова.
1. Старайтесь [убирать ненужные dependencies из Effects.](/learn/removing-effect-dependencies) Например, вместо мемоизации часто проще перенести объект или функцию внутрь Effect либо вынести их наружу.

Если конкретное взаимодействие всё ещё кажется тормозным, [используйте profiler из React Developer Tools](https://legacy.reactjs.org/blog/2018/09/10/introducing-the-react-profiler.html), чтобы понять, какие компоненты больше всего выигрывают от мемоизации, и добавьте мемоизацию там, где она нужна. Эти принципы в любом случае делают компоненты проще для отладки и понимания, так что следовать им полезно в любом случае. В долгосрочной перспективе мы исследуем [автоматическую granular memoization](https://www.youtube.com/watch?v=lGEMwh32soc), чтобы решить это раз и навсегда.

</Callout>

#### Пропуск повторного вычисления с `useMemo` {/*skipping-recalculation-with-usememo*/}

В этом примере реализация `filterTodos` **искусственно замедлена**, чтобы вы увидели, что происходит, когда какая-то JavaScript-функция, вызываемая во время рендера, действительно медленная. Попробуйте переключать вкладки и менять theme.

Переключение вкладок ощущается медленным, потому что оно заставляет замедленную `filterTodos` выполниться снова. Это ожидаемо, потому что `tab` изменился, а значит, всё вычисление *должно* выполниться заново. (Если интересно, почему оно выполняется дважды, это объясняется [здесь.](#my-calculation-runs-twice-on-every-re-render))

Поменяйте theme. **Благодаря `useMemo`, всё работает быстро несмотря на искусственное замедление!** Медленный вызов `filterTodos` был пропущен, потому что и `todos`, и `tab` (которые вы передаёте как dependencies в `useMemo`) не изменились с прошлого рендера.

```js tab="Tab 1"
//App.js
import { useState } from 'react';
import { createTodos } from './utils.js';
import TodoList from './TodoList.js';

const todos = createTodos();

export default function App() {
  const [tab, setTab] = useState('all');
  const [isDark, setIsDark] = useState(false);
  return (
    <>
      <button onClick={() => setTab('all')}>
        All
      </button>
      <button onClick={() => setTab('active')}>
        Active
      </button>
      <button onClick={() => setTab('completed')}>
        Completed
      </button>
      <br />
      <label>
        <input
          type="checkbox"
          checked={isDark}
          onChange={e => setIsDark(e.target.checked)}
        />
        Dark mode
      </label>
      <hr />
      <TodoList
        todos={todos}
        tab={tab}
        theme={isDark ? 'dark' : 'light'}
      />
    </>
  );
}
```

```js tab="Tab 3"
//utils.js
export function createTodos() {
  const todos = [];
  for (let i = 0; i < 50; i++) {
    todos.push({
      id: i,
      text: "Todo " + (i + 1),
      completed: Math.random() > 0.5
    });
  }
  return todos;
}

export function filterTodos(todos, tab) {
  console.log('[ARTIFICIALLY SLOW] Filtering ' + todos.length + ' todos for "' + tab + '" tab.');
  let startTime = performance.now();
  while (performance.now() - startTime < 500) {
    // Do nothing for 500 ms to emulate extremely slow code
  }

  return todos.filter(todo => {
    if (tab === 'all') {
      return true;
    } else if (tab === 'active') {
      return !todo.completed;
    } else if (tab === 'completed') {
      return todo.completed;
    }
  });
}
```

### Пропуск повторного рендера компонентов {/*skipping-re-rendering-of-components*/}

В некоторых случаях `useMemo` также помогает оптимизировать производительность повторного рендера дочерних компонентов. Чтобы показать это, предположим, что этот компонент `TodoList` передаёт `visibleTodos` как prop дочернему компоненту `List`:

```js
export default function TodoList({ todos, tab, theme }) {
  // ...
  return (
    <div className={theme}>
      <List items={visibleTodos} />
    </div>
  );
}
```

Вы заметили, что переключение prop `theme` на мгновение подвешивает приложение, но если убрать `<List />` из JSX, всё работает быстро. Это говорит о том, что стоит попробовать оптимизировать компонент `List`.

**По умолчанию, когда компонент повторно рендерится, React рекурсивно повторно рендерит всех его потомков.** Поэтому, когда `TodoList` повторно рендерится с другим `theme`, компонент `List` *тоже* повторно рендерится. Это нормально для компонентов, которым не требуется много вычислений при повторном рендере. Но если вы убедились, что повторный рендер происходит медленно, вы можете заставить `List` пропускать повторный рендер, когда его props такие же, как и в прошлый раз, обернув его в [`memo`:](/reference/react/memo)

```js
import { memo } from 'react';

const List = memo(function List({ items }) {
  // ...
});
```

**После этого изменения `List` будет пропускать повторный рендер, если все его props *такие же*, как и при предыдущем рендере.** Именно здесь и становится важным кэширование вычисления! Представьте, что вы вычисляли `visibleTodos` без `useMemo`:

```js 
export default function TodoList({ todos, tab, theme }) {
  // Every time the theme changes, this will be a different array...
  const visibleTodos = filterTodos(todos, tab);
  return (
    <div className={theme}>
      {/* ... so List's props will never be the same, and it will re-render every time */}
      <List items={visibleTodos} />
    </div>
  );
}
```

**В приведённом выше примере функция `filterTodos` всегда создаёт *другой* массив,** подобно тому, как литерал объекта `{}` всегда создаёт новый объект. Обычно это не проблема, но здесь это означает, что props `List` никогда не будут одинаковыми, и ваша оптимизация с [`memo`](/reference/react/memo) не сработает. Вот где `useMemo` оказывается полезным:

```js 
export default function TodoList({ todos, tab, theme }) {
  // Tell React to cache your calculation between re-renders...
  const visibleTodos = useMemo(
    () => filterTodos(todos, tab),
    [todos, tab] // ...so as long as these dependencies don't change...
  );
  return (
    <div className={theme}>
      {/* ...List will receive the same props and can skip re-rendering */}
      <List items={visibleTodos} />
    </div>
  );
}
```

**Обернув вычисление `visibleTodos` в `useMemo`, вы гарантируете, что между повторными рендерами оно будет иметь *одно и то же* значение** (пока dependencies не изменятся). Вам *не обязательно* оборачивать вычисление в `useMemo`, если только это не нужно по какой-то конкретной причине. В этом примере причина в том, что вы передаёте результат компоненту, обёрнутому в [`memo`,](/reference/react/memo) и это позволяет ему пропускать повторный рендер. Ниже на этой странице описаны и другие причины, по которым может понадобиться `useMemo`.

<Callout>

#### Memoization отдельных JSX-узлов {/*memoizing-individual-jsx-nodes*/}

```js 
export default function TodoList({ todos, tab, theme }) {
  const visibleTodos = useMemo(() => filterTodos(todos, tab), [todos, tab]);
  const children = useMemo(() => <List items={visibleTodos} />, [visibleTodos]);
  return (
    <div className={theme}>
      {children}
    </div>
  );
}
```

Поведение было бы таким же. Если `visibleTodos` не изменились, `List` не будет повторно рендериться.

JSX-узел вроде `<List items={visibleTodos} />` — это объект вида `{ type: List, props: { items: visibleTodos } }`. Создание такого объекта очень дешёвое, но React не знает, одинаково ли его содержимое по сравнению с прошлым разом. Поэтому по умолчанию React будет повторно рендерить компонент `List`.

Однако если React увидит абсолютно тот же JSX, что и при предыдущем рендере, он не будет пытаться повторно рендерить ваш компонент. Это потому, что JSX-узлы являются [immutable.](https://en.wikipedia.org/wiki/Immutable_object) Объект JSX-узла не может измениться со временем, поэтому React знает, что безопасно пропустить повторный рендер. Но чтобы это сработало, узел должен *действительно быть тем же самым объектом*, а не просто выглядеть так же в коде. Именно это и делает `useMemo` в данном примере.

Ручное оборачивание JSX-узлов в `useMemo` неудобно. Например, так нельзя сделать условно. Обычно именно поэтому компоненты оборачивают в [`memo`](/reference/react/memo), а не JSX-узлы.

</Callout>

### Предотвращение слишком частого запуска Effect {/*preventing-an-effect-from-firing-too-often*/}

Иногда вам может понадобиться использовать значение внутри [Effect:](/learn/synchronizing-with-effects)

```js {4-7,10}
function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');

  const options = {
    serverUrl: 'https://localhost:1234',
    roomId: roomId
  }

  useEffect(() => {
    const connection = createConnection(options);
    connection.connect();
    // ...
```
Это создаёт проблему. [Каждое reactive value должно быть объявлено как dependency вашего Effect.](/learn/lifecycle-of-reactive-effects#react-verifies-that-you-specified-every-reactive-value-as-a-dependency) Однако если объявить `options` как dependency, Effect будет постоянно переподключаться к чату:

```js 
  useEffect(() => {
    const connection = createConnection(options);
    connection.connect();
    return () => connection.disconnect();
  }, [options]); // 🔴 Problem: This dependency changes on every render
  // ...
```

Чтобы решить это, можно обернуть объект, который вам нужно использовать внутри Effect, в `useMemo`:

```js 
function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');

  const options = useMemo(() => {
    return {
      serverUrl: 'https://localhost:1234',
      roomId: roomId
    };
  }, [roomId]); // ✅ Only changes when roomId changes

  useEffect(() => {
    const connection = createConnection(options);
    connection.connect();
    return () => connection.disconnect();
  }, [options]); // ✅ Only changes when options changes
  // ...
```
Это гарантирует, что объект `options` будет одинаковым между повторными рендерами, если `useMemo` вернёт закэшированный объект.

Однако, поскольку `useMemo` — это оптимизация производительности, а не семантическая гарантия, React может отбросить закэшированное значение, если [для этого есть конкретная причина](#caveats). Это тоже приведёт к повторному запуску effect, **поэтому ещё лучше убрать необходимость в зависимости от объекта**, переместив объект *внутрь* Effect:

```js
function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');

  useEffect(() => {
    const options = { // ✅ No need for useMemo or object dependencies!
      serverUrl: 'https://localhost:1234',
      roomId: roomId
    }

    const connection = createConnection(options);
    connection.connect();
    return () => connection.disconnect();
  }, [roomId]); // ✅ Only changes when roomId changes
  // ...
```
Теперь ваш код стал проще и больше не нуждается в `useMemo`. [Подробнее об удалении dependencies Effect.](/learn/removing-effect-dependencies#move-dynamic-objects-and-functions-inside-your-effect)


### Memoization зависимости другого Hook {/*memoizing-a-dependency-of-another-hook*/}

Предположим, у вас есть вычисление, которое зависит от объекта, созданного прямо в теле компонента:

```js 
function Dropdown({ allItems, text }) {
  const searchOptions = { matchMode: 'whole-word', text };

  const visibleItems = useMemo(() => {
    return searchItems(allItems, searchOptions);
  }, [allItems, searchOptions]); // 🚩 Caution: Dependency on an object created in the component body
  // ...
```
Зависимость от такого объекта сводит memoization на нет. Когда компонент повторно рендерится, весь код внутри тела компонента выполняется снова. **Строки кода, создающие объект `searchOptions`, тоже будут выполняться при каждом повторном рендере.** Поскольку `searchOptions` является dependency вызова `useMemo`, а каждый раз оно другое, React понимает, что dependencies изменились, и пересчитывает `searchItems` каждый раз.

Чтобы исправить это, можно замемоизировать сам объект `searchOptions` до того, как передавать его как dependency:

```js 
function Dropdown({ allItems, text }) {
  const searchOptions = useMemo(() => {
    return { matchMode: 'whole-word', text };
  }, [text]); // ✅ Only changes when text changes

  const visibleItems = useMemo(() => {
    return searchItems(allItems, searchOptions);
  }, [allItems, searchOptions]); // ✅ Only changes when allItems or searchOptions changes
  // ...
```
В приведённом выше примере, если `text` не изменился, объект `searchOptions` тоже не изменится. Однако ещё лучше — перенести объявление объекта `searchOptions` *внутрь* функции вычисления `useMemo`:

```js 
function Dropdown({ allItems, text }) {
  const visibleItems = useMemo(() => {
    const searchOptions = { matchMode: 'whole-word', text };
    return searchItems(allItems, searchOptions);
  }, [allItems, text]); // ✅ Only changes when allItems or text changes
  // ...
```
Теперь ваше вычисление напрямую зависит от `text` (а это строка, и она не может «случайно» стать другой).

### Memoization функции {/*memoizing-a-function*/}

Предположим, что компонент `Form` обёрнут в [`memo`.](/reference/react/memo) Вы хотите передать ему функцию как prop:

```js
export default function ProductPage({ productId, referrer }) {
  function handleSubmit(orderDetails) {
    post('/product/' + productId + '/buy', {
      referrer,
      orderDetails
    });
  }

  return <Form onSubmit={handleSubmit} />;
}
```
Точно так же, как `{}` создаёт новый объект, объявления функций вроде `function() {}` и выражения вроде `() => {}` создают *другую* функцию при каждом повторном рендере. Само по себе создание новой функции — не проблема. Этого не нужно избегать! Однако если компонент `Form` замемоизирован, вероятно, вы хотите пропускать его повторный рендер, когда props не изменились. Prop, который *всегда* отличается, сведёт на нет смысл memoization.

Чтобы замемоизировать функцию с помощью `useMemo`, ваша функция вычисления должна будет возвращать другую функцию:

```js 
export default function Page({ productId, referrer }) {
  const handleSubmit = useMemo(() => {
    return (orderDetails) => {
      post('/product/' + productId + '/buy', {
        referrer,
        orderDetails
      });
    };
  }, [productId, referrer]);

  return <Form onSubmit={handleSubmit} />;
}
```
Это выглядит громоздко! **Memoization функций встречается настолько часто, что у React есть для этого встроенный Hook. Оборачивайте функции в [`useCallback`](/reference/react/useCallback) вместо `useMemo`**, чтобы не писать дополнительную вложенную функцию:

```js
export default function Page({ productId, referrer }) {
  const handleSubmit = useCallback((orderDetails) => {
    post('/product/' + productId + '/buy', {
      referrer,
      orderDetails
    });
  }, [productId, referrer]);

  return <Form onSubmit={handleSubmit} />;
}
```
Оба примера выше полностью эквивалентны. Единственное преимущество `useCallback` в том, что он позволяет не писать дополнительную вложенную функцию внутри. Больше он ничего не делает. [Подробнее о `useCallback`.](/reference/react/useCallback)

## Troubleshooting {/*troubleshooting*/}

### Моя вычисляемая функция запускается дважды при каждом повторном рендере {/*my-calculation-runs-twice-on-every-re-render*/}

В [Strict Mode](/reference/react/StrictMode) React будет вызывать некоторые ваши функции дважды вместо одного раза:

```js 
function TodoList({ todos, tab }) {
  // This component function will run twice for every render.

  const visibleTodos = useMemo(() => {
    // This calculation will run twice if any of the dependencies change.
    return filterTodos(todos, tab);
  }, [todos, tab]);

  // ...
```
Это ожидаемо и не должно ломать ваш код.

Такое поведение **только в development** помогает вам [держать компоненты чистыми.](/learn/keeping-components-pure) React использует результат одного вызова и игнорирует результат другого. Пока ваши компоненты и функции вычисления чистые, это не должно влиять на логику. Но если они случайно нечистые, это поможет заметить и исправить ошибку.

Например, вот нечистая функция вычисления, которая мутирует массив, полученный как prop:

```js 
  const visibleTodos = useMemo(() => {
    // 🚩 Mistake: mutating a prop
    todos.push({ id: 'last', text: 'Go for a walk!' });
    const filtered = filterTodos(todos, tab);
    return filtered;
  }, [todos, tab]);
```

React вызывает вашу функцию дважды, поэтому вы заметите, что todo добавляется два раза. Ваше вычисление не должно изменять существующие объекты, но изменять любые *новые* объекты, созданные во время вычисления, можно. Например, если функция `filterTodos` всегда возвращает *другой* массив, вы можете изменить именно *этот* массив:

```js 
  const visibleTodos = useMemo(() => {
    const filtered = filterTodos(todos, tab);
    // ✅ Correct: mutating an object you created during the calculation
    filtered.push({ id: 'last', text: 'Go for a walk!' });
    return filtered;
  }, [todos, tab]);
```
Читайте [keeping components pure](/learn/keeping-components-pure), чтобы узнать больше о чистоте.

Также посмотрите руководства по [обновлению объектов](/learn/updating-objects-in-state) и [обновлению массивов](/learn/updating-arrays-in-state) без мутаций.

### Мой вызов `useMemo` должен возвращать объект, но возвращает undefined {/*my-usememo-call-is-supposed-to-return-an-object-but-returns-undefined*/}

Этот код не работает:

```js 
  // 🔴 You can't return an object from an arrow function with () => {
  const searchOptions = useMemo(() => {
    matchMode: 'whole-word',
    text: text
  }, [text]);
```

В JavaScript `() => {` начинает тело стрелочной функции, поэтому фигурная скобка `{` не является частью вашего объекта. Из-за этого объект не возвращается, и возникают ошибки. Исправить это можно, добавив скобки `({` и `})`:

```js 
  // This works, but is easy for someone to break again
  const searchOptions = useMemo(() => ({
    matchMode: 'whole-word',
    text: text
  }), [text]);
```
Однако это всё ещё сбивает с толку и слишком легко ломается, если кто-то уберёт скобки.

Чтобы избежать этой ошибки, явно пишите `return`:

```js 
  // ✅ This works and is explicit
  const searchOptions = useMemo(() => {
    return {
      matchMode: 'whole-word',
      text: text
    };
  }, [text]);
```

### Каждый раз при рендере компонента вычисление в `useMemo` запускается заново {/*every-time-my-component-renders-the-calculation-in-usememo-re-runs*/}

Убедитесь, что вы указали массив dependencies вторым аргументом!

Если забыть массив dependencies, `useMemo` будет пересчитывать значение каждый раз:

```js 
function TodoList({ todos, tab }) {
  // 🔴 Recalculates every time: no dependency array
  const visibleTodos = useMemo(() => filterTodos(todos, tab));
  // ...
```
Это исправленная версия с массивом dependencies во втором аргументе:

```js 
function TodoList({ todos, tab }) {
  // ✅ Does not recalculate unnecessarily
  const visibleTodos = useMemo(() => filterTodos(todos, tab), [todos, tab]);
  // ...
```
Если это не помогает, значит проблема в том, что хотя бы одна из dependencies отличается от предыдущего рендера. Отладить это можно, вручную выводя dependencies в консоль:

```js
  const visibleTodos = useMemo(() => filterTodos(todos, tab), [todos, tab]);
  console.log([todos, tab]);
```
Затем можно щёлкнуть правой кнопкой по массивам из разных повторных рендеров в консоли и выбрать для обоих «Store as a global variable». Предположим, первый массив сохранился как `temp1`, а второй как `temp2`; тогда в консоли браузера можно проверить, одинаково ли каждое значение dependencies в обоих массивах:

```js
Object.is(temp1[0], temp2[0]); // Is the first dependency the same between the arrays?
Object.is(temp1[1], temp2[1]); // Is the second dependency the same between the arrays?
Object.is(temp1[2], temp2[2]); // ... and so on for every dependency ...
```
Когда вы найдёте dependency, которая ломает memoization, либо найдите способ убрать её, либо [замемоизируйте её тоже.](#memoizing-a-dependency-of-another-hook)

### Мне нужно вызывать `useMemo` для каждого элемента списка в цикле, но это запрещено {/*i-need-to-call-usememo-for-each-list-item-in-a-loop-but-its-not-allowed*/}

Предположим, что компонент `Chart` обёрнут в [`memo`](/reference/react/memo). Вы хотите пропускать повторный рендер каждого `Chart` в списке, когда повторно рендерится компонент `ReportList`. Однако вы не можете вызывать `useMemo` в цикле:

```js 
function ReportList({ items }) {
  return (
    <article>
      {items.map(item => {
        // 🔴 You can't call useMemo in a loop like this:
        const data = useMemo(() => calculateReport(item), [item]);
        return (
          <figure key={item.id}>
            <Chart data={data} />
          </figure>
        );
      })}
    </article>
  );
}
```

Вместо этого вынесите компонент для каждого элемента и memoize данные отдельно для каждого элемента:

```js 
function ReportList({ items }) {
  return (
    <article>
      {items.map(item =>
        <Report key={item.id} item={item} />
      )}
    </article>
  );
}

function Report({ item }) {
  // ✅ Call useMemo at the top level:
  const data = useMemo(() => calculateReport(item), [item]);
  return (
    <figure>
      <Chart data={data} />
    </figure>
  );
}
```

Как вариант, можно вообще убрать `useMemo` и вместо этого обернуть сам `Report` в [`memo`.](/reference/react/memo) Если prop `item` не изменяется, `Report` пропустит повторный рендер, а значит и `Chart` тоже пропустит повторный рендер:

```js 
function ReportList({ items }) {
  // ...
}

const Report = memo(function Report({ item }) {
  const data = calculateReport(item);
  return (
    <figure>
      <Chart data={data} />
    </figure>
  );
});
```
